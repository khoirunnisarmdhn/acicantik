Use case: Login
Aktor: Admin, Administrasi, Keuangan, Pimpinan
Deskripsi: Proses autentikasi pengguna untuk masuk ke dalam sistem
Pre-Condition: Aktor membuka halaman login sistem
Post-Condition: Sistem mengarahkan pengguna ke halaman dashboard sesuai hak aksesnya
Skenario Normal
AktorSistem
1. Aktor membuka halaman login.
2. Sistem menampilkan form login dengan field email dan password.
3. Aktor memasukkan email dan password yang valid.
4. Aktor menekan tombol "Masuk ke Sistem".
5. Sistem memverifikasi data dan mengarahkan pengguna ke halaman dashboard sesuai hak akses.
Skenario Alternatif: Email Tidak Terdaftar
AktorSistem
1. Aktor membuka halaman login.
2. Sistem menampilkan form login.
3. Aktor memasukkan email yang tidak terdaftar.
4. Aktor menekan tombol "Masuk ke Sistem".
5. Sistem menampilkan pesan error "These credentials do not match our records".
Skenario Alternatif: Password Salah
AktorSistem
1. Aktor membuka halaman login.
2. Sistem menampilkan form login.
3. Aktor memasukkan password yang salah.
4. Aktor menekan tombol "Masuk ke Sistem".
5. Sistem menampilkan pesan error "These credentials do not match our records".

Use case: Mengelola User & Role
Aktor: Admin
Deskripsi: Proses menambah, mengubah, dan menghapus data pengguna beserta hak aksesnya
Pre-Condition: Aktor berhasil login sebagai admin dan membuka menu User Management
Post-Condition: Sistem menyimpan data user beserta hak aksesnya dan menampilkan hasil masukan dari admin
Skenario Normal
AktorSistem
1. Aktor menekan tombol tambah user.
2. Sistem menampilkan form tambah user.
3. Aktor mengisi username, nama lengkap, email, memilih hak akses, dan mengisi password.
4. Aktor menekan tombol Simpan.
5. Sistem menyimpan data user dan hak aksesnya, menampilkan pesan "User dan hak akses berhasil dibuat!" dan menampilkan data user pada tabel.
Skenario Alternatif: Password Tidak Cocok
AktorSistem
1. Aktor menekan tombol tambah user.
2. Sistem menampilkan form tambah user.
3. Aktor mengisi password dan konfirmasi password yang berbeda.
4. Aktor menekan tombol Simpan.
5. Sistem tidak menyimpan data dan menampilkan pesan error "konfirmasi password tidak cocok".
Skenario Alternatif: Email Sudah Terdaftar
AktorSistem
1. Aktor menekan tombol tambah user.
2. Sistem menampilkan form tambah user.
3. Aktor memasukkan email yang sudah terdaftar.
4. Aktor menekan tombol Simpan.
5. Sistem tidak menyimpan data dan menampilkan pesan error "email sudah terdaftar".

Use case: Mengelola Data Proyek
Aktor: Admin, Administrasi
Deskripsi: Proses menambah, mengubah, dan menghapus data proyek
Pre-Condition: Aktor berhasil login dan membuka menu Data Proyek
Post-Condition: Sistem menyimpan data proyek, termin otomatis terbuat, dan menampilkan hasil masukan dari aktor
Skenario Normal
AktorSistem
1. Aktor menekan tombol tambah proyek.
2. Sistem menampilkan form tambah proyek.
3. Aktor mengisi nama proyek, memilih client, mengisi nilai kontrak, jumlah termin (3 atau 4), tanggal mulai, tanggal selesai, dan status.
4. Aktor menekan tombol Simpan.
5. Sistem menyimpan data proyek, membuat termin secara otomatis (DP 20%, Termin tengah, Termin Akhir 10%), dan menampilkan pesan sukses.
Skenario Alternatif: Jumlah Termin Tidak Valid
AktorSistem
1. Aktor menekan tombol tambah proyek.
2. Sistem menampilkan form tambah proyek.
3. Aktor mengisi jumlah termin dengan nilai di luar range 3–4.
4. Aktor menekan tombol Simpan.
5. Sistem tidak menyimpan data dan menampilkan pesan validasi.
Skenario Alternatif: Nama Proyek Tidak Diisi
AktorSistem
1. Aktor menekan tombol tambah proyek.
2. Sistem menampilkan form tambah proyek.
3. Aktor mengosongkan field nama proyek.
4. Aktor menekan tombol Simpan.
5. Sistem tidak menyimpan data dan menampilkan pesan error "Tidak boleh kosong".

Use case: Mengelola Termin Proyek
Aktor: Admin, Administrasi
Deskripsi: Proses mengubah data termin proyek yang telah dibuat otomatis oleh sistem
Pre-Condition: Aktor berhasil login dan membuka menu Termin Proyek
Post-Condition: Sistem memperbarui data termin dan menghitung nominal otomatis berdasarkan persentase × nilai kontrak
Skenario Normal
AktorSistem
1. Aktor menekan tombol edit pada salah satu termin.
2. Sistem menampilkan form edit termin proyek.
3. Aktor mengubah due date dan keterangan dengan data valid.
4. Aktor menekan tombol Simpan.
5. Sistem menyimpan perubahan data termin, menghitung nominal otomatis, dan menampilkan pesan sukses.
Skenario Alternatif: Persentase Melebihi 100%
AktorSistem
1. Aktor menekan tombol edit pada salah satu termin.
2. Sistem menampilkan form edit termin proyek.
3. Aktor mengubah persentase sehingga total semua termin dalam proyek melebihi 100%.
4. Aktor menekan tombol Simpan.
5. Sistem tidak menyimpan data dan menampilkan pesan error "Total persentase melebihi 100%! Sisa yang tersedia: X%".

Use case: Mengelola Transaksi Kas Masuk & Kas Keluar
Aktor: Admin, Keuangan
Deskripsi: Proses mencatat transaksi penerimaan kas dan pengeluaran kas secara otomatis menghasilkan jurnal debit dan kredit
Pre-Condition: Aktor berhasil login dan membuka menu Transaksi
Post-Condition: Sistem menyimpan transaksi kas, jurnal otomatis terbuat, dan status termin diperbarui
Skenario Normal: Kas Masuk
AktorSistem
1. Aktor membuka menu Transaksi → Kas Masuk dan menekan tombol tambah.
2. Sistem menampilkan form tambah kas masuk dengan no form otomatis.
3. Aktor mengisi tanggal, memilih kategori, proyek, termin, metode bayar, nominal, keterangan, dan mengupload bukti transfer.
4. Aktor menekan tombol Simpan.
5. Sistem menyimpan transaksi kas masuk, mengubah status termin menjadi "Lunas", membuat jurnal otomatis (debit dan kredit), dan menampilkan pesan sukses.
Skenario Normal: Kas Keluar
AktorSistem
1. Aktor membuka menu Transaksi → Kas Keluar dan menekan tombol tambah.
2. Sistem menampilkan form tambah kas keluar dengan no form otomatis.
3. Aktor mengisi tanggal, memilih kategori, proyek, vendor, metode bayar, mengisi rincian item, dan keterangan.
4. Aktor menekan tombol Simpan.
5. Sistem menyimpan transaksi kas keluar dengan total nominal rincian, menyimpan rincian item, membuat jurnal otomatis, dan menampilkan pesan sukses.
Skenario Alternatif: Bukti Transfer Tidak Diupload (Kas Masuk)
AktorSistem
1. Aktor mengisi form kas masuk tanpa mengupload bukti transfer.
2. Aktor menekan tombol Simpan.
3. Sistem tidak menyimpan data dan menampilkan pesan error "bukti transfer wajib diupload".
Skenario Alternatif: Rincian Kosong (Kas Keluar)
AktorSistem
1. Aktor mengisi form kas keluar tanpa mengisi rincian item.
2. Aktor menekan tombol Simpan.
3. Sistem tidak menyimpan data dan menampilkan pesan error "rincian minimal 1 item wajib diisi".
Skenario Alternatif: Pengeluaran Melebihi Anggaran LRA
AktorSistem
1. Aktor mengisi form kas keluar dengan nominal yang melebihi anggaran LRA kategori tersebut.
2. Aktor menekan tombol Simpan.
3. Sistem menyimpan data tetapi menampilkan pesan peringatan "PERINGATAN: Pengeluaran kategori melebihi anggaran LRA" beserta selisih kelebihan anggaran.

Use case: Melihat Laporan Realisasi Anggaran
Aktor: Admin, Keuangan, Pimpinan
Deskripsi: Proses menampilkan laporan perbandingan anggaran dan realisasi pengeluaran per kategori berdasarkan proyek
Pre-Condition: Aktor berhasil login dan membuka menu Laporan → LRA Proyek
Post-Condition: Sistem menampilkan tabel perbandingan anggaran vs realisasi beserta selisih per kategori
Skenario Normal
AktorSistem
1. Aktor membuka menu Laporan → LRA Proyek.
2. Sistem menampilkan halaman laporan LRA dengan dropdown pilihan proyek.
3. Aktor memilih proyek dari dropdown.
4. Aktor menekan tombol Lihat Laporan.
5. Sistem menampilkan tabel perbandingan anggaran vs realisasi per kategori dilengkapi dengan kolom selisih.
Skenario Alternatif: Proyek Tidak Dipilih
AktorSistem
1. Aktor membuka menu Laporan → LRA Proyek tanpa memilih proyek.
2. Aktor menekan tombol Lihat Laporan.
3. Sistem tidak menampilkan data tabel, halaman menunggu pemilihan proyek.

Use case: Melihat Laporan Jurnal Umum
Aktor: Admin, Keuangan, Pimpinan
Deskripsi: Proses menampilkan laporan jurnal umum berisi entri debit dan kredit dari seluruh transaksi kas berdasarkan periode
Pre-Condition: Aktor berhasil login dan membuka menu Laporan → Jurnal Umum
Post-Condition: Sistem menampilkan daftar jurnal sesuai periode yang dipilih
Skenario Normal
AktorSistem
1. Aktor membuka menu Laporan → Jurnal Umum.
2. Sistem menampilkan halaman jurnal umum dengan filter bulan dan tahun (default bulan berjalan).
3. Aktor memilih filter bulan dan tahun.
4. Aktor menekan tombol Filter.
5. Sistem menampilkan daftar jurnal dengan kolom tanggal, no referensi, kode akun, nama akun, debit, dan kredit sesuai periode yang dipilih.
Skenario Alternatif: Periode Tidak Ada Transaksi
AktorSistem
1. Aktor membuka menu Jurnal Umum.
2. Sistem menampilkan halaman jurnal umum.
3. Aktor memilih bulan dan tahun yang tidak memiliki transaksi.
4. Aktor menekan tombol Filter.
5. Sistem menampilkan pesan "Data jurnal periode tersebut tidak ditemukan" dan mengarahkan kembali ke halaman jurnal.
